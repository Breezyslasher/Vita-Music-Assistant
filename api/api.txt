## Connection Methods

### WebSocket API (Recommended)
URL: ws://<server>:8095/ws
- Real-time bidirectional communication
- Automatic event notifications
- Requires message_id field for request/response matching

### HTTP API (RPC)
URL: POST http://<server>:8095/api
- Simple RPC-like interface
- message_id is optional
- Headers: Authorization: Bearer <token>, Content-Type: application/json

### Server Info (No Auth)
GET http://<server>:8095/info

## Authentication

### Get Login Providers
GET http://<server>:8095/auth/providers

### Login with Credentials
POST http://<server>:8095/auth/login
{
  "credentials": {
    "username": "<username>",
    "password": "<password>"
  }
}
Response: { "success": true, "token": "<access_token>", "user": {...} }

### Login with Specific Provider (e.g., Home Assistant OAuth)
POST http://<server>:8095/auth/login
{
  "provider_id": "homeassistant",
  "credentials": {...provider-specific...}
}

### First-Time Setup (only if no users exist)
POST http://<server>:8095/setup
{
  "username": "admin",
  "password": "secure_password"
}

### WebSocket Authentication (REQUIRED as first command)
{
  "message_id": "auth-123",
  "command": "auth",
  "args": { "token": "<access_token>" }
}
Response: { "message_id": "auth-123", "result": { "authenticated": true, "user": {...} } }

### Token Types
- Short-lived tokens: Created during login. Expire after 30 days of inactivity, auto-renew on use.
- Long-lived tokens: Created via auth/token/create. Expire after 10 years. For integrations.

### Player Events
player_added            - New player discovered

queue_added             - New queue created

## Remote Access (WebRTC)

### Get Remote Access Info
{
  "message_id": "remote-123",
  "command": "remote_access/info",
  "args": {}
}
Response: {
  "enabled": true,
  "connected": true,
  "remote_id": "MA-XXXX-XXXX",
  "signaling_url": "wss://signaling.music-assistant.io/ws"
}

### How WebRTC Remote Access Works
1. Client gets Remote ID from remote_access/info command
2. Connect to signaling server at wss://signaling.music-assistant.io/ws
3. WebRTC Data Channel establishes encrypted P2P connection through NAT/firewalls
4. Remote commands work identically to local WebSocket API
5. End-to-end encryption via DTLS/SRTP
6. Requires Home Assistant Cloud subscription for STUN/TURN servers

## Sendspin Streaming Protocol

Sendspin is Music Assistant's audio streaming protocol for client-side playback.

### Flow
1. Client requests stream URL: player_queues/get_stream_url
2. Server returns Sendspin WebSocket URL (ws:// or wss://)
3. Client connects to Sendspin endpoint
4. Server sends control messages (JSON) and audio frames (binary)
5. Client decodes and plays audio locally

### Control Messages (JSON)
{ "type": "audio_format", "codec": "flac", "sample_rate": 44100, "channels": 2, "bits_per_sample": 16 }
{ "type": "stream_start" }
{ "type": "stream_end" }
{ "type": "metadata", "key": "title", "value": "Song Name" }
{ "type": "error", "message": "Error description" }

### Audio Codecs Supported
- PCM (raw audio)
- FLAC (lossless)
- Opus (lossy, efficient)

## Best Practices
- Use WebSocket API for real-time applications
- Handle connection drops and reconnect automatically
- Subscribe to relevant events instead of polling
- Use unique message IDs for WebSocket commands
- Implement proper error handling
- Don't poll the REST API frequently (use WebSocket events)
- Don't hardcode server URLs (make them configurable)
